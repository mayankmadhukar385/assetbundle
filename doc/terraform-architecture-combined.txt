
  ═══════════════════════════════════════════════════════════════════════
              APPROACH 1: PARAMETERIZED (Single Root Module)
  ═══════════════════════════════════════════════════════════════════════

  Description:
  In this approach, a single set of Terraform configuration files (root
  module) is shared across all environments (dev, uat, prod). Environment-
  specific values are supplied through separate .tfvars files (e.g.
  dev.tfvars, uat.tfvars, prod.tfvars). The CI/CD pipeline selects the
  appropriate .tfvars file at runtime using the -var-file flag or
  Terraform workspaces. This keeps the codebase DRY — there is only one
  copy of main.tf, variables.tf, etc. — and any infrastructure change
  automatically applies to all environments. State isolation is achieved
  via workspaces or distinct backend keys per environment.

  ═══════════════════════════════════════════════════════════════════════
              APPROACH 2: SEPARATE FOLDER (Per Environment)
  ═══════════════════════════════════════════════════════════════════════

  Description:
  In this approach, each environment (dev, uat, prod) has its own
  dedicated folder containing a full set of Terraform files (main.tf,
  variables.tf, backend.tf, etc.). Each folder independently calls
  shared reusable modules from a common modules/ directory. Because
  every environment has its own backend.tf, state files are naturally
  isolated — eliminating the risk of one environment accidentally
  modifying another's resources. This approach is ideal when environments
  need to diverge in configuration or when strict blast-radius control
  is required, at the cost of some code duplication across folders.

  ═══════════════════════════════════════════════════════════════════════
     APPROACH 3: DYNAMIC TERRAFORM CODE GENERATION FROM METADATA
  ═══════════════════════════════════════════════════════════════════════

  Description:
  Instead of writing Terraform .tf files by hand, you define your
  infrastructure in a simple metadata file (YAML / JSON), and a code
  generator (Python + Jinja2, Go, Jsonnet, etc.) reads that metadata
  and AUTOMATICALLY GENERATES the Terraform .tf files. Developers only
  edit the metadata — never the generated .tf files. This is like a
  "template engine for infrastructure" and is ideal for large-scale
  environments with many similar services (e.g. 50 microservices each
  needing EC2 + RDS). Adding a new service is just a few lines of YAML.
